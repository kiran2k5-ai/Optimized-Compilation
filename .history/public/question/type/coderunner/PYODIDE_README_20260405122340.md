# Pyodide Integration for CodeRunner

## Overview

This integration enables **client-side Python code execution** using **Pyodide.js** directly in the browser, without sending code to the server.

## Key Features

✅ **No Server Submission** - Code never leaves the browser  
✅ **Instant Feedback** - Results displayed immediately  
✅ **Browser-based Execution** - Uses Pyodide.js (Python runtime in WASM)  
✅ **Blocks Form Submission** - Prevents any fallback to server  
✅ **Real-time Results Display** - Shows pass/fail for each test case  

## How It Works

### Flow
```
Student clicks Check button
    ↓
Pyodide check button interceptor captures click
    ↓
Form submission BLOCKED
    ↓
Student code extracted from textarea
    ↓
Pyodide executes code in browser with test cases
    ↓
Results displayed in UI (no server communication)
```

## Files Created/Modified

### New Files Created:

1. **`amd/src/pyodide_executor.js`**
   - Core Pyodide execution engine
   - Loads Pyodide from CDN
   - Executes Python code with test cases
   - Captures output and results

2. **`amd/src/pyodide_checkbutton.js`**
   - Intercepts Check button clicks
   - Prevents form submission
   - Calls Pyodide executor
   - Displays results in UI

3. **`pyodide_styles.css`**
   - Styles for test results display
   - Alert styling
   - Test case formatting

### Files Modified:

1. **`renderer.php`**
   - Added CSS loading for Pyodide styles
   - Added JavaScript AMD calls to initialize Pyodide check button
   - Added form submission blocking

2. **`question.php`**
   - Added `$usepyodide` property for future configuration

## Usage

The Pyodide integration is **automatically active** for all CodeRunner questions:

1. Student enters Python code in the answer textarea
2. Student clicks the "Check" button
3. Code executes in **Pyodide** (no server call)
4. Results display immediately in the browser
5. **Form is never submitted to server**

## Test Case Format

Test cases must be available in the DOM. The system looks for:

1. Hidden input field: `input[name*="testcases_<questionId>"]` containing JSON
2. DOM elements with class `coderunner-test-case` and:
   - `.test-input` - contains input data
   - `.test-output` - contains expected output

## Example Test Case (JSON format)

```json
[
  {
    "testNumber": 1,
    "input": "5",
    "expectedOutput": "25"
  },
  {
    "testNumber": 2,
    "input": "10",
    "expectedOutput": "100"
  }
]
```

## Limitations

### ⚠️ Current Limitations:

1. **File I/O** - No file system access (Pyodide runs in sandbox)
2. **External Libraries** - Limited to Python standard library + pre-bundled packages
3. **Initial Load** - First load slower (~3-5 seconds for Pyodide to load from CDN)
4. **Memory** - Browser memory constraints limit large data processing
5. **No Networking** - Can't make HTTP requests or access external APIs
6. **Security** - Code runs in browser context (see security note below)

### Languages Supported:
- ✅ **Python** (via Pyodide)
- ❌ Other languages not supported (Java, C++, PHP, etc.)

## Security Considerations

⚠️ **IMPORTANT**: This system executes code in the **browser**, not a secured sandbox on the server.

### Security Risks:
- Student can inspect test cases via DevTools
- Student can modify code or results before viewing
- No true isolation from system
- Suitable for **PRACTICE ONLY**, NOT graded assessments

### Recommendations:

**DO USE for:**
- ✅ Practice exercises (non-graded)
- ✅ Learning environments
- ✅ Tutoring sessions
- ✅ Self-assessment tools

**DON'T USE for:**
- ❌ Graded exams
- ❌ High-stakes assessments
- ❌ Final evaluations
- ❌ Academic integrity-sensitive contexts

### Alternative for Graded Work:
Use a **hybrid approach**: Practice with Pyodide, but final submission uses server-side execution with signature verification.

## Performance

### Speed:
- **First load**: ~3-5 seconds (Pyodide downloads from CDN)
- **Cached loads**: ~0.5-1 second (browser cache)
- **Code execution**: Variable (depends on code complexity)
- **Simple test**: <100ms
- **Complex test**: 1-5 seconds

### Optimization Tips:
- Cache Pyodide after first load
- Keep test cases small
- Avoid heavy computations
- Use CDN for fast delivery

## Testing & Debugging

### Browser DevTools Testing:

1. Open Developer Tools (F12)
2. Go to Console tab
3. Check for messages like:
   - `✓ Pyodide loaded successfully`
   - `Initialization Pyodide check button`
   - `Found X test cases, ready to execute with Pyodide`

### Test Execution Logs:

The console will show:
```
Check button clicked - using Pyodide execution
Execution results: { success: true|false, testResults: [...] }
```

## Troubleshooting

### Issue: Pyodide fails to load
**Solution**: Check CDN connectivity and browser console for errors

### Issue: Test cases not found
**Solution**: Ensure test cases are properly embedded in DOM or hidden input

### Issue: Code execution times out
**Solution**: Simplify the code or reduce test case complexity

### Issue: ImportError for packages
**Solution**: Only standard library packages supported (NumPy, Pandas, etc. not available)
